从一个最简单的 case 说起
用户心里在打分
一个人在搜索框敲下："预算 3000 想给爸妈换手机，要屏幕大、续航久、好上手，选哪款？"
他要的不是参数表
而是和我处境一样的人（给老人买、图省心）真实买过什么、踩过哪些坑——借真实成交和评价，做自己的购买决策。
系统一两秒内做三件事
召回真实成交 / 评价 / 导购笔记 → LLM 组织成贴合处境的推荐 → 呈现；他在心里默默打分，再用脚投票（下单 / 加购 / 退货）。
那个"分"我们看不见、没法直接优化——只能用一个模型去近似它，这就是 Reward Model。

Rubric 是什么
维度 × 档位，不是一个抽象分
把评判拆成多条 criteria，每条配一个明确判定档，而非拍一个抽象分。
① 维度 = 该看哪几条（商品推荐）
  • 是否按处境（肤质 / 机型 / 预算 / 尺码）匹配
  • 是否基于真实销量 / 评价，而非刷单 / 软广
  • 有没有避坑 / 适用边界（缺货、版本差异）
  • 看完能直接下单，还是"自己再看看"
② 档位 = 每条怎么判
Green：完全符合，可采纳
Yellow：方向对，细节有瑕疵
Red：错误或有害，必须拦截
Underconstrained：信息不足，应主动说"不确定"
换到医疗：criteria = 是否核对剂量 / 禁忌、有无漏红旗、是否提示就医、信息不足是否 abstain；档位同样 Green / Yellow / Red / Underconstrained。
维度 × 档位拼出失败的坐标：错在哪条、错到哪一档。

GSB 是没办法的办法
能拆成 Rubric 的，尽量别停在 GSB
GSB（好 / 平 / 差两两对比）只给整体偏好，不告诉你好在哪、错在哪——是最后才用的兜底。
为什么会卡在 GSB
  • 大概率不是事情不可拆，是缺这个领域的专家
  • 例："美"——只会说这个好看那个不好看
  • 没人能说清标准，就只能两两对比
有专家，再主观也能拆成 Rubric
  • 真实专家总能给出 criteria：「美」美在哪
  • 如：白底干净 / 45°斜角构图 / 留白比例…
  • 每条再配 Green / Yellow / Red 档位
想尽办法把 GSB 往 Rubric 转，是评测体系最重要的事之一——GSB 只该是临时兜底，不是终点。

效果度量（四）· 多维 merge
别用加权平均
一条错误的建议，再流畅、结构再好，总分也不该高。
1
硬约束（一票否决）
电商：刷单虚假好评、错价错参、推无货、停售品；医疗：误导用药、漏红旗、超范围下结论——分数直接压下去。
2
场景调权
电商抬真实评价 / 适配 / 性价比；医疗抬安全性 / 循证 / 就医提示；按品类 / 科室再微调权重。
3
叠主观分
丰富度、结构、表达、简洁——锦上添花，但永远在硬约束之后。
线性加权是最差的选择：它会让"漂亮的错误"捞到高分。

Failure Code
把"用户的不满"结构化
电商：问"预算 3000 给爸妈换手机"却推一水儿"网红爆款"、还都是年轻人在用 → 软广未识别 + 人群未匹配。

Code 是"现象"，不是"根因"
"软广"的根因可能在召回（捞进商单）、prompt（没要求排广告）、或 RM（没有软广维度）——要再往下归因。
Code 的三个用处
  • 组织评测集：用户眼里的问题解没解
  • 组织训练集：专做对抗样本
  • 做效果掉了：指标掉了哪个 Code 在作怪
好 Code：可观察、判定一致、与效果直接相关、能拿来训 RM。

怎么把模型推向这把尺子
SFT → 蒸馏 → DPO → RL
一条主线：从模仿标准答案，到模仿强教师，到对齐"为什么 A 比 B 好"，再到自己探索更优。
方式怎么调优点缺点 / 坑
SFT 监督微调模仿标准答案简单、快、好落地易背题、过时会带偏、不会自我探索
蒸馏师徒 Best-of-N，学优质输出高效学好行为天花板 ≈ 教师
DPO 直接偏好优化用 pairwise（A:B）优化，隐式 RM工程简单、稳、省一个 RM需两两、OOD 泛化差、偏爱长
RL（RLHF/PPO/GRPO）用 RM 做 reward，在线探索代替背题探索、对齐、追配多探贵、不稳、reward hacking
训练还有什么：RLAIF（AI 反馈替代人标）、过程奖励 PRM（给推理每一步打分）、KTO / ORPO（偏好优化变体）、课程学习 / hard example 重采样。

训练管线
先分清基模「有没有」，再决定怎么训
别一上来就训。先问：这个能力基模到底有没有——答案决定用哪种训练方式。
1
过滤已具备
基模已经会的别花训练预算重看——先测出来，从训练集里踢掉。
2
高采样探边界
对剩下的大量采样（pass@k / Best-of-N）：能到就到 → 只是激发 / 稳定差；怎么都不到 → 真不具备。
3
分而治之
激发 / 稳定问题 → RL 直接拉齐；真不具备 → SFT 冷启动注入种子，再交给 RL。
SFT 只用于 cold start：除冷启动外不做 SFT——SFT 偏记忆、会侵蚀基模已有泛化，提升与稳定都交给 RL。

硬蒸馏
一步拿到顶级能力，代价是打崩原生推理——不推荐裸用
把 SOTA 老师模型（或多模型投票）的"推理 → 结果"整段拽出来，直接 SFT 进基模。
好处
  • 一步到位：瞬间接近老师的顶级能力
  • 工程极简、见效最快
坏处（为什么不推荐裸用）
  • 强灌老师 token，基模原生推理被打崩
  • 学到"形"丢了"神"，OOD 一塌
  • 后续 RL 也难救
怎么既拿能力又保住推理？三条伤害更小的路径：① 少量 SFT 做 cold start + RL 激发（RL 本质是把 pass@k 拉成 pass@1）；② 自补 CoT：让待训基模用自己的推理方式补链，而非老师的；③ 两者结合：cold start 起步 → 自采样 + 自补 CoT → RL 精修，全程少灌老师原始 token。

On-Policy 蒸馏（OPD）
卡在 SFT 和 RL 中间的第三条路
让学生先按自己策略生成轨迹，教师在学生真实会到达的状态上给稠密反馈——纠正的正好是它自己真犯错的地方。
定位：模仿与 RL 的中间层
  • 比 SFT 更贴近推理时真实分布（学自己的轨迹）
  • 比 RL 的稀疏 outcome reward 给得更稠密（逐 token 反馈）
正好补上「硬蒸馏崩推理」
  • 硬蒸馏：在教师理想路径上学 → 打崩原生推理
  • OPD：在学生自己轨迹上学 → 保住推理方式
  • 即上页「保留推理」的正式范式
工业拐点（Qwen3）：同样从 8B checkpoint 出发，OPD 在数学 / 代码上优于直接 RL，且 GPU 小时约只要 1/10。

OPD 最该记的几点 + 入门路线

GRPO
用「一组答案互相比」当基线——便宜、稳、为推理而生
为什么这么火：省掉和策略一样大的价值网络，显存腰斩；可验证奖励 + GRPO = DeepSeek-R1 的推理配方。
1
原理
同一道题采一组 G 个答案，用组内奖励均值当基线：Â_i =（r_i − mean(r)）/ std(r)；省掉 PPO 的 critic。
2
关键参数
组大小 G=8~64；KL 系数 β 是安全带；clip ε；采样温度。
3
最佳实践
组内归一化奖励；奖励可验证（数学 / 代码）最看；G 大 → 稳但贵。
一句话：把"绝对打分"换成"组内相对名次"，省一个网络、天然稳，特别吃可验证奖励。

升级到底换来什么

Workflow 优劣
  • ＋路径确定、好调试、好兜底、行为可审计
  • －每个新场景都要重写一遍
  • －分支越加越乱，改一处动全身
  • －撞上没到的情况直接抓瞎

  • ＋能力可组合，新场景已有 skill 拼
  • ＋随基模变强自然涨，不重复主干
  • ＋渐进加载，省上下文
  • －吃基模能力，弱模型乱调；路径不定，兜底更难

落地 · 确定性知识
用「确定」的 medwiki 替代「不确定」的 RAG
循证问答的可信，不能建立在「概率检索」之上——这是方法论「知识层 / LLM Wiki」那页在医疗场景的正式形态。
RAG · 不确定，难复现
  • 检索是概率匹配：切块 / embedding / topK 一变，命中即变
  • 拼接上下文再生成，答案漂移、不可复现
  • 来源难追溯，易产生「看似有据」的幻觉
medwiki · 确定，可回放
  • 基于 llmwiki（结构化循证知识库）+ agentic search（沿引用逐跳检索）
  • 条目、证据等级、引用可寻址，沿确定路径检索
  • 可复现、可回放、可溯源；更新 = 改条目，有版本、可回归——像代码一样管理
确定性来自结构化知识与可寻址引用，而非一次概率检索——与「指南即代码」同源。
